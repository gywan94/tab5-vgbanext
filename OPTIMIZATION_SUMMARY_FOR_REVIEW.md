# Tab5-VGBANext GBA 模拟器性能优化总结（供专家评审）

> 目的：把已做过的所有优化尝试 + 实测数据 + 物理约束整理清楚，请教**还有哪些提升空间**。

---

## 1. 硬件 / 软件背景

| 项 | 配置 |
|---|---|
| 板子 | M5Stack Tab5 |
| SoC | **ESP32-P4**，双 RISC-V HP 核，**顺序、单发射**（in-order, single-issue） |
| 主频 | **360 MHz**（400MHz 选项在本板/IDF 组合下会回落到 90MHz，无法用） |
| 内存 | 32MB PSRAM（Octal SPI，DDR 200MHz）+ **768KB 片上 SRAM**（其中 **256KB 配为 L2 cache**） |
| 显示 | 720×1280 MIPI-DSI，PPA 硬件缩放/旋转 |
| 模拟核 | **VBA-Next**（libretro/vba-next），纯**解释器**，表派发（4096 项 ARM 函数指针表 / THUMB 类似） |
| ISA | 模拟 ARM7TDMI（ARM + THUMB 两种模式） |

测试 ROM：**Yoshi Sample**（公认 ARM-heavy 最坏例，~41% ARM / ~58% THUMB），koudai/口袋妖怪（THUMB-heavy，典型例）。

## 2. 当前性能基线（实测）

| 游戏 | 解释器 FPS | 备注 |
|---|---|---|
| Yoshi（ARM 最坏例） | **26–28**（base）/ **35**（开双核渲染） | CPU ~37ms/帧 |
| koudai（典型 THUMB） | **46**（base）/ **54**（双核渲染） | |
| sizhijian | 41 / — | |

**CPI（每条模拟指令的主机周期）≈ 183**（聚合：总周期/帧 ÷ 总指令/帧，硬件实测）。
对比 PC（x86 乱序核）跑同一核 ≈ 28 CPI → **慢 ~6.5×，主要是顺序核 + cache 停顿**。

## 3. 核心瓶颈：每条指令 183 周期花在哪

`armExecute` 主循环每条指令的工作（已逐行核对源码）：

| 桶 | 内容 | 主机指令数 | 性质 |
|---|---|---|---|
| ① 固定派发开销 | 取指/预取记账/查表/**间接调用**/周期记账/循环判断 | ~48 条 | 不管什么指令都要付 |
| ② handler 本体 | 桶形移位 + 运算 + NZCV 标志 + 写回 | 12–35 条 | 因指令而异 |
| ③ 访存停顿 | `bus.reg[]`/标志/PC 每条从内存/L2 重载（load-use stall） | 贯穿 | **顺序核的主要停顿源** |
| ④ 间接派发预测失败 | `(*armInsnTable[idx])(opcode)` 每次冲刷流水线 | — | 顺序核失预测代价高 |

**关键结论：~48/60 主机指令是"派发开销"，真正"算这条指令"只占 12–35 条。大半成本在"把指令派发出去"，不在计算本身。**

### 逐指令周期估算（固定 48 + handler，× 顺序核 ~2.5 周期/主机指令）

| ARM 指令 | 主机指令 | ≈ 周期 |
|---|---|---|
| MOV（无标志） | ~60 | ~150 |
| ADDS（带标志+移位） | ~78 | ~190 |
| LDR（访存） | ~83 + 内存读 | ~210+ |
| 条件不成立的 op | ~48 | ~120 |

## 4. 已尝试的所有优化（带实测数据）

### A. 有效、已采用 ✅

| 优化 | 收益 | 说明 |
|---|---|---|
| `USE_TWEAKS=1`（空闲循环加速） | 19→24 fps | VBA 自带 speedhack |
| **双核分离**（PPU 渲染→core1，CPU→core0） | Yoshi **+25%**(28→35)，koudai **+17%**(46→54) | 唯一的真双核并行 |
| `-O3` | 优于 -O2 | 巨大的 CPULoop 偏好 -O3 |
| **DMA 批量 memmove**（RAM→RAM 无副作用走 memcpy） | **+2-3%** | 命中 PSRAM 80MB/s 顺序带宽 |
| **预取记账门控**（IWRAM 代码跳过 gamepak 预取空算） | **+3%** | 安全、对所有游戏 IWRAM 代码生效 |

### B. 试过、零收益或负收益 ❌

| 优化 | 结果 | 原因 |
|---|---|---|
| `USE_TWEAK_SPEEDHACK=1` | **0** | USE_TWEAKS 已含空闲加速 |
| IWRAM/BIOS → 内部 SRAM（E2） | **0** | **L2 已把 PSRAM 数据喂到 14ns，搬了白搬** |
| 热循环函数 → IRAM（IRAM_ATTR） | **0** | **L2 已把 flash 取指罩住** |
| L2 cache 256→512KB | **链接失败** | 内部 SRAM 溢出 169KB（768KB 的饼分不出来） |
| 跳帧（frameskip） | 仅省 ~5ms | PPU 渲染本来就小（~5ms），瓶颈在 CPU |
| 帧间 memcpy 搬到 core1 | 0（甚至略差） | 256→240 strided vs 连续，PSRAM 上一样 |

### C. JIT / 动态重编译（research，默认 disarm）

| 尝试 | 结果 | 说明 |
|---|---|---|
| **THUMB JIT**（M0–M6，块链接） | **峰值 1.6×**（koudai 55→89fps/CPU 17→10ms） | 真实有效**但**：① 64KB 内部 exec arena 只装 ~113 块，漫游超出就回退到比解释器慢；② 音频在 armed 时坏（APU offset 漂）；③ 不可出货 |
| **ARM JIT**（M-A1..A5，本次新做） | **~0%**（26 vs 26fps，干净测量） | 翻译**逐位正确**（ARMJITVAL=0）、覆盖 58%、稳定；但 **ARM 条件执行**让每条条件 op 都要付运行时谓词守卫，抵消翻译收益 |
| JIT 码放 PSRAM（扩 arena） | **死路** | PSRAM **随机取指 672ns**，工作集一散就抖到 28fps |
| 块链接 vs 无链接 | 链接是 THUMB 1.6× 的关键；ARM 链接后仍 ~0% | ARM 的瓶颈是条件税 + 每块 C 往返 |

### D. Tier-3 解释器重写（computed-goto 线程化）—— 可行性已用 microbench 证伪

板上 microbench（同样 per-op 工作，两种派发方式）：
```
DISPATCH-BENCH @360MHz: funcptr=34.5 cyc/op  threaded=28.5 cyc/op  speedup=1.21x
```
- 线程化省 **6 周期/op（固定**：call/ret + 进出栈 + 标志从内存改寄存器）。
- microbench 的 op 极轻（34.5 周期）→ 显得 1.21×。
- **真实指令 183 周期/op，同样省 6 周期 → 仅 ~3-5%**。
- 拿不到教科书 1.5-2×，因为：**GBA 寄存器堆是运行时动态索引 `reg[(op>>4)&15]`，机器寄存器无法动态索引 → 必须留内存 → 占大头的访存停顿省不掉**。

**结论：数周重写换 ~3-5%，不值。**

## 5. 关键硬件测量数据（P4 @360MHz，实测）

| 测项 | 结果 |
|---|---|
| PSRAM 顺序读 | **80 MB/s**（18 周期/字） |
| PSRAM 随机读（冷，4MiB 打穿 L2） | **672 ns/次**（242 周期） |
| 内部 SRAM 随机读 | **14 ns/次**（5 周期） |
| L2 cache | 256KB（512KB 装不下） |
| CPI（聚合，解释执行） | **~183 周期/指令** |
| 线程化派发节省 | ~6 周期/指令（固定） |

**两次零收益实验（E2 IWRAM→SRAM、IRAM 热循环）共同证明：L2 对热工作集同时罩住了 PSRAM 数据 + flash 取指 → 模拟器不是内存瓶颈，是"顺序核解释执行"本身。**

## 6. 已确认的物理约束（为什么某些路走不通）

1. **第二个核不能帮 CPU**：模拟一个 CPU 是严格串行（指令 N+1 依赖 N 的结果，每 6 条还有分支等标志）。拆"译码核/执行核"被数据依赖锁死，跨核同步(~100-300ns)比省下的译码(~50ns)贵几倍。能并行的"CPU | 渲染+音频"已经做了。
2. **寄存器堆必须在内存**：动态索引，机器寄存器放不下 → 访存停顿无法消除。
3. **JIT 码必须在内部 SRAM**：PSRAM 随机取指 672ns 太慢；内部 exec SRAM 只 ~64KB（~113 块）。
4. **L2 给不到 512KB**：768KB 的饼要分给驱动/buffer/JIT arena，512KB L2 直接链接溢出。
5. **400MHz 不可用**：本板回落 90MHz。

## 7. 仍存疑 / 想请教专家的方向

> 以下是我**没做、或没把握**的，想问还有没有空间：

1. **gpSP 式分支补丁（block linking via direct branch patching）**：现在 JIT 块跑完都回 C 循环再查表派发（每块 ~40 主机指令往返税）。gpSP 把块 A 的尾分支**直接改写成跳块 B 的码**，消除往返。我估算这是 THUMB JIT 从 1.6× 再上台阶的关键，但要重构块 ABI（单一外层栈帧 + 块间直跳 + i-cache 一致性 + 块淘汰时反补丁）。**问：值得吗？ARM 的条件税会不会还是抵消？**

2. **惰性标志（lazy flag evaluation）**：ARM/THUMB 每个 S-op 都急切算 NZCV，但读标志的频率远低于写。延迟到"被条件读时才算"能省。但 VBA 的标志逻辑深度交织，重写易错。**问：在这种解释器上 ROI 如何？**

3. **换核**：gpSP（libretro/gpsp）本身带成熟的 ARM+THUMB dynarec（不是解释器）。**问：在 ESP32-P4 上移植 gpSP 的 dynarec，比继续优化 VBA 解释器更值吗？**（gpSP 同样面临 64KB 内部 SRAM 的 arena 约束。）

4. **PGO（profile-guided optimization）**：用代表性 ROM 训练后重编 VBA。嵌入式上要把 .gcda 经串口导出再重编，折腾。**问：典型能拿几个百分点？**

5. **P4 特有指令/特性**：`-march=rv32imafc_zicsr_zifencei_xesploop_xespv` 已带 Espressif 的 loop/PIE 扩展。**问：有没有被低估的 P4 硬件特性（比如 cache 预取提示、双发射？）能帮上？**

6. **降低 per-op 访存**：把最热的几个 GBA 寄存器（PC、SP、LR、CPSR）固定映射到主机寄存器（用全局寄存器变量 `register ... asm("x..")`），减少 reg[] 的重载？**问：在 RISC-V + GCC 下可行且有效吗？**

## 8. 一句话现状

> **在 ESP32-P4（顺序核 360MHz）上，VBA-Next 解释器跑 GBA 已被榨到接近物理极限：CPI≈183，廉价优化全部用完（双核渲染 +25%、DMA +3%、预取门控 +3% 是仅有的真收益）。L2 罩住了内存（PSRAM/flash 都不是瓶颈），第二个核帮不了 CPU（串行依赖），线程化重写经 microbench 证伪只值 ~3-5%。THUMB JIT 能到 1.6× 但受 64KB SRAM arena + 音频问题困，ARM JIT 因条件执行税 ~0%。真正可能的下一台阶是 gpSP 式块链接 / 换 gpSP dynarec / 惰性标志——都是数周级重工程，且都撞同一堵 64KB 内部 SRAM 墙。**
